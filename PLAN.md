A **single SPIRE server** is built which simplifies the architecture by removing the complexities of cross-domain federation. 
It places the entire security of an **OpenClaw Non-Human Persona (NHP)** on the integrity of that one "Root of Trust."

In this centralized model, the **Trust Bundle** is the "Source of Truth" that all agents and tools use to recognize each other. 
Here is the architectural refinement for that infrastructure.

---

### 1. The Anatomy of a Single-Server Trust Bundle
In a single-server setup, the Trust Bundle is relatively static but critical. It serves as the "public directory" of authorized signing keys.

* **Root CA Certificate:** The self-signed (or enterprise-signed) certificate of the SPIRE Server. This is the "Anchor."
* **Active Signing Keys:** The public keys the server is currently using to sign X.509-SVIDs for your OpenClaw agents.
* **Rotation Metadata:** Information that tells the agents how frequently to check for a new version of the bundle.

**The Cryptographic Relationship:**
The Trust Bundle ($B$) allows an agent to verify a presented identity ($SVID_{peer}$) by confirming that the signature ($\sigma$) was generated by a key ($K_{pub}$) currently present in the bundle:
$$Verify(SVID_{peer}, B) \implies \exists K_{pub} \in B \text{ such that } Verify(\sigma, K_{pub}) = \text{True}$$

---

### 2. The "Bootstrap" Handshake: Getting the Bundle to OpenClaw
Since an agent cannot "know" who to trust until it has the bundle, we use a **Local Attestation** flow. For an OpenClaw instance running on a Linux host, this is an out-of-band communication:

1.  **The Agent's Arrival:** The OpenClaw process starts and immediately looks for the SPIRE Agent's **Unix Domain Socket** (e.g., `/run/spire/sockets/workload.sock`).
2.  **The Interrogation:** The SPIRE Agent uses the kernel's `getpeereid` to identify the PID of the OpenClaw process. It verifies the **Unix Selector** (e.g., `unix:user:openclaw` and `unix:sha256:abcd...`).
3.  **Delivery of the Anchor:** Once the "Who" is verified, the SPIRE Agent sends the **Trust Bundle** over the secure socket directly into the OpenClaw memory space. 
4.  **No Secrets on Disk:** By keeping the bundle and SVIDs in-memory, you prevent a compromised disk from allowing an attacker to "clone" the NHP's identity.



---

### 3. Defining the "NHP" Registration Entry
On your single SPIRE Server, you must define exactly what constitutes a valid OpenClaw agent. This is done via a **Registration Entry**.

```bash
# Example 2026 SPIRE Registration for OpenClaw Finance Agent
spire-server entry create \
    -parentID spiffe://enterprise.com/spire/agent/linux-node-01 \
    -spiffeID spiffe://enterprise.com/nhp/openclaw/finance-auditor \
    -selector unix:uid:1001 \
    -selector unix:sha256:e3b0c442... \
    -ttl 300 \
    -admin
```
* **ParentID:** Binds the agent to a specific, verified physical or virtual machine.
* **Selectors:** Ensure that *only* the specific OpenClaw binary, running under the `finance` user, can claim this NHP identity.
* **TTL (300s):** Forces the agent to check back every 5 minutes. If you delete this entry, the agent's identity expires in 300 seconds.

---

### 4. Reverse Trust: Protecting the Agent from the World
The Trust Bundle isn't just for the server to verify the agent; it’s for the **Agent to verify the Tools**. 

If OpenClaw needs to call an **MCP (Model Context Protocol)** server to access a database, it uses its Trust Bundle to perform a **Mutual TLS (mTLS)** handshake.
* **The Check:** The agent asks the MCP server for its SVID.
* **The Verification:** The agent checks the MCP server's certificate against the **Trust Bundle**.
* **The Result:** If the MCP server was set up by a malicious actor (e.g., a "Shadow AI" tool), its certificate won't be signed by the SPIRE Root. OpenClaw, seeing the mismatch in its Trust Bundle, will **refuse to connect**.

---

### 5. Managing the Single Point of Failure
Since all NHP identities rely on this one SPIRE Server, your prototype must address the **Availability-Security Tradeoff**:

| Risk | Mitigation |
| :--- | :--- |
| **Server Offline** | Agents can't rotate SVIDs; they lose access in minutes. **Fix:** Use a HA (High Availability) SPIRE setup with a shared SQL database. |
| **Root Key Compromise** | An attacker can mint any NHP identity. **Fix:** Back the SPIRE Server with a **Hardware Security Module (HSM)** or a Cloud KMS. |
| **"Zombie" Identities** | A compromised agent continues to run until TTL expires. **Fix:** Implement an "Emergency Revocation" hook in your eBPF monitor. |



---

### Final Implementation Insight
By focusing on a single-server architecture, your **NHP Trust Bundle** acts as a "Digital Perimeter." Any tool, script, or agent that doesn't "speak SPIFFE" or isn't signed by your Root is physically invisible to the OpenClaw agent.
